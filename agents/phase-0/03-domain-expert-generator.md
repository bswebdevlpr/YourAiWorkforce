# Domain Expert Generator - Phase 0

> **역할**: PRD에서 도메인을 추출하여 Domain Expert 페르소나를 동적으로 생성
> **소요**: 30-45분
> **난이도**: ⭐⭐⭐⭐☆

## 📥 입력 파일

- ✅ **필수**: `docs/specs/prd.md` (Product Discovery Agent의 출력)
- ⚠️ **참고**: `docs/specs/architecture.md` (선택적)

---

## 🔨 작업 Step-by-Step

### Step 1: PRD에서 도메인 키워드 추출 (10분)

PRD의 **"문제 정의"**와 **"도메인 키워드"** 섹션을 읽고 핵심 도메인을 식별하세요.

**추출 패턴**:
```
"[Domain Object]를 [Action]하는" → 도메인: [Domain Object의 학문 분야]
```

**예시 매핑**:

| PRD 아이디어 | Domain Object | 추출된 도메인 | 하위 도메인 |
|------------|--------------|-------------|-----------|
| AI 음식 추천 | 음식, 영양 | 영양학, 요리학, 식품과학 | 영양소, 식이 제한, 맛 프로필 |
| 부동산 검색 | 부동산 | 부동산학, GIS, 도시 계획 | 부동산 평가, 공간 데이터, 용도지역 |
| 주식 분석 | 주식, 금융 | 금융공학, 경제학 | 기술적 분석, 재무제표, 시장 지표 |
| 운동 루틴 추천 | 운동, 피트니스 | 운동생리학, 스포츠과학 | 근육 그룹, 회복, 운동 강도 |

> PRD 부록의 도메인 키워드를 **기반으로 확장**한다. PDA가 추출한 키워드를 재추출하지 않고, 학문적·기술적 깊이를 더하는 방향으로 보강한다.

**작업**:
1. PRD를 읽고 Domain Object 식별
2. 그 객체가 속한 학문 분야 2-3개 나열
3. 각 학문 분야의 하위 도메인 5-10개 나열

**체크리스트**:
- [ ] 주 도메인 2-3개 식별 (예: nutrition science, urban planning)
- [ ] 각 도메인의 하위 도메인 5-10개 식별
- [ ] 도메인이 너무 일반적이지 않음 (예: "웹 개발" ❌, "영양소 분석" ✅)

---

### Step 2: 전문가 페르소나 구축 (10분)

식별된 도메인을 바탕으로 **가상의 전문가 페르소나**를 생성하세요.

**템플릿**:

```markdown
# Domain Expert: [이름] - [학위/자격증명]

**전문 분야**: [주 도메인]
**하위 전문 영역**: [5-10개 하위 도메인]
**경력**: [가상 경력 10-20년]
**주요 자격**:
- [관련 박사 학위 또는 전문 자격증]
- [관련 경험, 예: "15년간 XX 연구"]
- [출판 또는 프로젝트 경험]

---

## 핵심 지식 영역

### 1. [하위 도메인 1]
[이 영역의 핵심 개념 3-5개, 각 1-2줄 설명]

### 2. [하위 도메인 2]
[...]

[5-10개 하위 도메인 각각에 대해 반복]

---

## 데이터 소스 추천

프로젝트에 필요한 데이터를 얻을 수 있는 신뢰할 수 있는 소스:

1. **[소스 이름]**: [URL 또는 설명]
   - **형식**: [JSON, CSV, API, 웹 스크래핑 등]
   - **라이선스**: [공개 도메인, CC BY, MIT 등]
   - **커버리지**: [데이터 범위, 예: "50,000개 레시피"]
   - **품질**: [신뢰도 평가, 예: "정부 공식 통계 데이터"]

2. [추가 소스들...]

**최소 3-5개 데이터 소스 제공**

**데이터 소스 식별 프로세스**:
1. PRD의 P0 기능에 필요한 데이터 유형을 나열한다
2. 각 데이터 유형에 대해: 공개 API → 공개 데이터셋 → 학술 데이터베이스 → 웹 스크래핑 순으로 탐색한다
3. 각 소스에 라이선스, 포맷(JSON/CSV/SQL), 예상 레코드 수를 기록한다
4. 소스를 찾지 못한 데이터 유형은 "수동 생성 필요"로 표시한다

## 일반적인 함정과 엣지 케이스

도메인에서 자주 발생하는 오류나 간과하기 쉬운 케이스:

1. **[함정/엣지 케이스 1]**: [설명 + 해결 방법]
2. **[함정/엣지 케이스 2]**: [...]

[5-10개 나열]

---

## 품질 평가 루브릭

프로젝트 결과물의 도메인 정확성을 평가하는 기준:

| 카테고리 | 평가 기준 | 배점 |
|---------|----------|------|
| **데이터 정확도** | [구체적 기준, 예: "공식 출처 데이터와 일치"] | 30점 |
| **데이터 커버리지** | [예: "주요 엔티티의 80% 이상 포함"] | 25점 |
| **도메인 관례 준수** | [예: "업계 표준 분류 체계 사용"] | 20점 |
| **엣지 케이스 처리** | [예: "예외 데이터, 경계 조건 처리"] | 15점 |
| **문서화 품질** | [예: "데이터 출처 명시, 신뢰도 표시"] | 10점 |

**총점: 100점**
**합격 기준: 85점 이상** (A- 이상)

**평가 방법**: 각 카테고리의 점수는 다음 기준으로 산출한다:
- 만점: 해당 영역의 모든 데이터가 공식 출처와 일치하거나, 전문가 수준의 분석이 포함됨
- 80%: 대부분 정확하나 일부 누락 또는 미검증 항목 존재
- 60% 이하: 주요 데이터 오류 또는 핵심 영역 누락
```

> 위 템플릿의 구체성 수준을 목표로 한다. 각 섹션이 빈 플레이스홀더가 아니라 도메인에 맞는 실질적 내용으로 채워져야 한다.

**체크리스트**:
- [ ] 전문가 이름과 자격증명 작성
- [ ] 5-10개 하위 도메인 각각에 대해 핵심 개념 설명
- [ ] 최소 3-5개 데이터 소스 추천 (라이선스 명시)
- [ ] 5-10개 도메인 함정/엣지 케이스 나열
- [ ] 100점 만점 평가 루브릭 작성

### Step 3: Domain Knowledge Document 작성 (15분)

Step 2에서 구축한 페르소나를 바탕으로 **참조 문서**를 작성하세요.

이 문서는 다른 모든 에이전트(특히 Data Engineer, Algorithm Engineer, QA Lead)가 참조할 "도메인 백과사전" 역할을 합니다.

**목표 분량**: 300-500줄

**구조**:
```markdown
# Domain Knowledge: [도메인 이름]

**작성일**: [날짜]
**작성자**: Domain Expert Generator
**전문가**: [Step 2에서 생성한 페르소나 이름]

---

## 0. Executive Summary
[이 도메인의 핵심을 5줄 이내로 요약]

---

## 1. 도메인 배경 및 역사
[이 도메인의 배경, 발전 과정, 중요성]
[예: 도메인의 발전 과정, 주요 전환점]

---

## 2. 핵심 개념 및 용어
[Step 2의 "핵심 지식 영역"을 확장]
[각 개념을 5-10줄로 상세히 설명]

---

## 3. 데이터 구조 및 분류 체계
[도메인에 자연스러운 데이터 모델]
[예: 음식 → 영양소, 재료, 레시피 관계]
[예: 부동산 → 위치, 가격, 용도지역 관계]

---

## 4. 데이터 소스 및 품질 기준
[Step 2의 "데이터 소스 추천"을 확장]
[각 소스의 장단점, 사용 시 주의사항]

---

## 5. 알고리즘 및 계산 방법
[도메인 특화 알고리즘 추천]
[예: 영양 점수 계산, 맛 프로필 매칭]
[예: 부동산 가치 평가, 유사 매물 추천]

---

## 6. 품질 벤치마크 및 표준
[Step 2의 "품질 평가 루브릭"을 확장]
[도메인에서 인정받는 품질 기준]

---

## 7. 알려진 예외 사항 및 엣지 케이스
[Step 2의 "일반적인 함정"을 확장]
[각 엣지 케이스를 상세히 설명하고 해결 방법 제시]

---

## 8. 참고 문헌 및 추가 자료
[도메인 학습을 위한 책, 논문, 웹사이트]
```

**체크리스트**:
- [ ] 300줄 이상 작성 (충분히 상세한 참조 자료)
- [ ] 모든 8개 섹션 작성 완료
- [ ] 구체적인 예시 포함 (추상적 설명만 ❌)
- [ ] 다른 에이전트가 이 문서만 읽고 도메인 이해 가능한 수준

## 💡 참고: 도메인별 예시

### 예시 1: 음식 추천
- **도메인**: 영양학, 요리학
- **전문가**: Chef Kim Nutrition (요리 영양학자)
- **핵심 지식**: 영양소, 맛 프로필, 식이 제한, 알레르겐
- **데이터 소스**: USDA FoodData Central, Open Food Facts, Spoonacular
- **엣지 케이스**: 알레르겐 교차 오염, 영양소 조리 손실, 맛 주관성

### 예시 2: 부동산 검색
- **도메인**: 부동산학, GIS, 도시 계획
- **전문가**: Agent Park Property (공인 부동산 전문가)
- **핵심 지식**: 부동산 평가, 용도지역, 학군, 시장 지표
- **데이터 소스**: 공개 부동산 기록, Census 데이터, OpenStreetMap
- **엣지 케이스**: 용도지역 변경, 학군 경계 불일치, 지오코딩 정확도

## 🔍 품질 검증

Domain Knowledge Document가 충분한지 확인하는 방법:

**자가 테스트**:
1. 이 문서만 읽고 데이터베이스 스키마를 설계할 수 있는가?
2. 이 문서만 읽고 데이터 소스를 찾을 수 있는가?
3. 이 문서만 읽고 알고리즘 정확도를 평가할 수 있는가?
4. 이 문서만 읽고 엣지 케이스를 처리할 수 있는가?

**모두 "예"라면 통과. 하나라도 "아니오"라면 해당 섹션을 보강하세요.**

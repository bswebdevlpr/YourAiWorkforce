# 기획자 (Product Discovery) - Phase 0

> **역할**: 원시 사용자 아이디어를 구조화된 제품 요구사항으로 변환
> **소요**: 35-45분
> **난이도**: ⭐⭐⭐☆☆

## 📥 입력

- ✅ **필수**: 사용자 아이디어 (자유 형식 텍스트)
  - 예: "AI 기반 음식 추천 앱"
  - 예: "부동산 검색 웹사이트를 만들고 싶어"
- ⚠️ **선택**: 제약사항 (예산 범위, 타임라인, 대상 사용자층)

> ⚠️ 입력에 구체적 예산 금액, API 키, 비밀번호 등 민감 정보를 포함하지 않는다.

---

## 🔨 작업 Step-by-Step

### Step 1: 아이디어 파싱 및 도메인 키워드 추출 (5분)

아이디어 문자열을 다음 요소로 분해한다:

```
[Action] + [Domain Object] + [Platform]
```

**예시**: "AI 기반 음식을 **추천**하는 **앱**" → 추천(Action), 음식(Domain Object), 앱(Platform)

또한 도메인 전문가 생성(DEA)에 필요한 키워드를 추출한다:

- **기술 도메인**: [예: Web Community Platform, Content Curation, Recommendation Engine]
- **학문 도메인**: [예: Nutrition Science, Urban Planning, Fandom Studies]
- **하위 도메인**: [예: User-generated Content, Gamified Learning, Dietary Restrictions]

**예외 처리**:

- Action이 식별되지 않는 경우: 사용자에게 핵심 기능을 질문한다
- Domain Object가 너무 광범위한 경우: 범위 축소를 요청한다 (예: "음식" → "한식 레시피")
- Platform이 명시되지 않은 경우: 웹을 기본값으로 가정하되, 사용자에게 확인한다

**자율 실행 시 기본값**:

- Action이 식별되지 않는 경우: 도메인의 가장 일반적인 Action(검색, 조회, 비교 중 택1)을 가정한다
- Domain Object가 너무 광범위한 경우: PRD 작성 후 사용자 확인 시점에서 조정한다
- Platform이 명시되지 않은 경우: 웹(Web)을 기본값으로 사용한다

**체크리스트**:

- [ ] 핵심 동사(Action) 식별됨
- [ ] 도메인 객체(Domain Object) 명확함
- [ ] 플랫폼(Platform) 결정됨
- [ ] 도메인 키워드 추출됨 (기술 도메인 + 학문 도메인 + 하위 도메인)

---

### Step 2: 타겟 사용자 페르소나 생성 (10분)

**3-5개의 사용자 페르소나**를 작성한다. 각 페르소나는 다음 형식을 따른다:

```markdown
### 페르소나 1: [이름]

- **배경**: [나이, 직업]
- **기술 수준**: [디지털 리터러시 수준, 주 사용 기기]
- **목표**: 이 제품으로 무엇을 달성하고 싶은가?
- **페인 포인트**: 현재 어떤 문제를 겪고 있는가?
- **사용 시나리오**: 언제, 어디서, 어떻게 사용할 것인가?
```

**체크리스트**:

- [ ] 최소 3개 페르소나 작성
- [ ] 각 페르소나가 서로 다른 사용 목적 가짐
- [ ] 기술 수준 다양성 (초보자 ~ 전문가)

> 💬 사용자 확인: "타겟 사용자 페르소나를 작성했어요. 검토해주세요."

---

### Step 3: 기능 식별 및 우선순위 분류 (15분)

**8-15개 기능**을 식별하고 다음 카테고리로 분류한다. 입력에 예산/타임라인 제약이 있으면 P0 범위를 이에 맞게 조정한다.

**P0 범위 조정 기준**:

- 타임라인 2주 이하: P0 기능 최대 2개
- 타임라인 2-4주: P0 기능 최대 4개
- 타임라인 4주 이상 또는 미지정: P0 기능 최대 5개

| 우선순위 | 정의                                       |
| -------- | ------------------------------------------ |
| **P0**   | MVP 필수. 이것 없으면 제품이 작동하지 않음 |
| **P1**   | Critical. 출시 전 필수                     |
| **P2**   | Nice-to-have. 출시 후 추가 가능            |
| **P3**   | Future. 장기 로드맵                        |

기능 목록은 반드시 다음 형식으로 작성한다:

```markdown
### P0 (MVP 필수)

1. **[기능명]**: [1-2줄 설명] (대상: [페르소나명])

### P1 (Critical)

2. **[기능명]**: [1-2줄 설명] (대상: [페르소나명])

### P2 (Nice-to-have)

...

### P3 (Future)

...
```

**체크리스트**:

- [ ] P0 기능 3-5개 (너무 많으면 범위 축소 고려)
- [ ] P1 기능 3-7개
- [ ] P2/P3 기능 합쳐서 5-10개
- [ ] 각 기능이 페르소나의 목표와 연결됨
- [ ] 페르소나 간 목표 충돌 시 P0 사용자(첫 번째 페르소나)를 우선한다
- [ ] 제약사항(예산/타임라인)이 P0 범위에 반영됨

> 💬 사용자 확인: "기능 우선순위 분류를 완료했어요. P0 기능이 적절한지 확인해주세요."

---

### Step 4: 측정 가능한 성공 지표 정의 (5분)

**3-5개의 정량적 메트릭**을 정의한다:

```markdown
## 성공 지표

1. **[메트릭명]**: [목표값]
   - 측정 플랫폼: [P0 범위 내 플랫폼]
   - 측정 방법: [어떻게 측정할 것인가]
   - 임계값: [최소 허용 값]
```

**체크리스트**:

- [ ] 모든 메트릭이 숫자로 측정 가능
- [ ] 각 메트릭에 명확한 임계값 설정
- [ ] 측정 방법이 구체적으로 정의됨

---

### Step 5: 범위 경계 정의 (5분)

**명시적으로 포함하지 않을 것**을 정의한다:

```markdown
### 범위 내 (In Scope)

- ✅ [포함되는 기능/도메인]

### 범위 외 (Out of Scope)

- ❌ [제외되는 기능/도메인]
```

**체크리스트**:

- [ ] 범위 내 항목 3-5개
- [ ] 범위 외 항목 3-5개 (향후 혼란 방지)
- [ ] 도메인 경계가 명확함

---

### Step 6: Phase 마일스톤 초안 작성 (5분)

P0-P3 기능 분류를 기반으로 Phase별 마일스톤 초안을 작성한다. 이 마일스톤은 아키텍트가 아키텍처 문서에서 구체화한다.

```markdown
## Phase 마일스톤 (예상)

- **Phase 0 완료**: PRD + 아키텍처 + 도메인 지식 문서
- **Phase 1 완료**: 데이터베이스 + 기본 API + 빌드 설정
- **Phase 2 완료**: 모든 P0 기능 작동
- **Phase 3 완료**: P1 기능 + 알고리즘 구현
- **Phase 4 완료**: SEO + 성능 최적화 + i18n
- **Phase 5 완료**: 도메인 전문가 검증 통과
- **Phase 6 완료**: 프로덕션 배포
```

**체크리스트**:

- [ ] 각 Phase의 완료 조건이 PRD 기능과 연결됨

# Product Discovery Agent (PDA) - Phase 0

> **역할**: 원시 사용자 아이디어를 구조화된 제품 요구사항으로 변환
> **소요**: 30-45분
> **난이도**: ⭐⭐⭐☆☆

## 📥 입력

**사용자 아이디어** (자유 형식 텍스트):
- 예시: "한중일 한자를 비교하는 웹사이트를 만들고 싶어"
- 예시: "AI 기반 음식 추천 앱"
- 예시: "부동산 검색 플랫폼"

**선택적 제약사항**:
- 예산 제한
- 타임라인
- 대상 사용자층

---

## 🔨 작업 Step-by-Step

### Step 1: 아이디어 파싱 (5분)

아이디어 문자열을 다음 요소로 분해하세요:

```
[Action] + [Domain Object] + [Platform]
```

**예시**:
- "한중일 한자를 **비교**하는 **웹사이트**" → 비교(Action), 한자(Domain Object), 웹(Platform)
- "AI 기반 **음식 추천** **앱**" → 추천(Action), 음식(Domain Object), 모바일(Platform)

**체크리스트**:
- [ ] 핵심 동사(Action) 식별됨
- [ ] 도메인 객체(Domain Object) 명확함
- [ ] 플랫폼(Platform) 결정됨

---

### Step 2: 타겟 사용자 페르소나 생성 (10분)

**3-5개의 사용자 페르소나**를 만드세요. 각 페르소나는:

```markdown
### 페르소나 1: [이름]
- **배경**: [나이, 직업, 기술 수준]
- **목표**: 이 제품으로 무엇을 달성하고 싶은가?
- **페인 포인트**: 현재 어떤 문제를 겪고 있는가?
- **사용 시나리오**: 언제, 어디서, 어떻게 사용할 것인가?
```

**TriHanzi 예시**:
1. **학습자 (Lee, 25세 한국인 일본어 학습자)**: 일본 한자와 한국 한자의 차이를 배우고 싶음
2. **교사 (Wang, 35세 중국어 교사)**: 학생들에게 CJK 문자 차이를 시각적으로 설명할 자료 필요
3. **언어학자 (Tanaka, 45세 비교 언어학 연구자)**: 한중일 문자 변형 패턴 연구를 위한 데이터베이스 필요

**체크리스트**:
- [ ] 최소 3개 페르소나 작성
- [ ] 각 페르소나가 서로 다른 사용 목적 가짐
- [ ] 기술 수준 다양성 (초보자 ~ 전문가)

---

### Step 3: 기능 식별 및 우선순위 분류 (15분)

**8-15개 기능**을 식별하고 다음 카테고리로 분류하세요:

| 우선순위 | 정의 | 예시 |
|---------|------|------|
| **P0** | MVP 필수. 이것 없으면 제품이 작동하지 않음 | 한자 기본 정보 표시, 검색 |
| **P1** | Critical. 출시 전 필수 | 3개국 비교 테이블, False Friends |
| **P2** | Nice-to-have. 출시 후 추가 가능 | 교육 레벨 표시, PDF 내보내기 |
| **P3** | Future. 장기 로드맵 | 사용자 계정, 플래시카드 생성 |

**템플릿**:

```markdown
## P0 (MVP 필수)
1. **[기능명]**: [1-2줄 설명]
2. **[기능명]**: [1-2줄 설명]

## P1 (Critical)
3. **[기능명]**: [1-2줄 설명]

## P2 (Nice-to-have)
...

## P3 (Future)
...
```

**TriHanzi 실증 예시**:
- **P0**: 문자 기본 정보 표시, 검색, 한중일 발음 표시
- **P1**: 비교 테이블, False Friends 감지, 다국어 지원
- **P2**: 교육 레벨(HSK/JLPT), PDF 내보내기, 유사 문자 추천
- **P3**: 사용자 계정, 학습 진도 추적, 플래시카드

**체크리스트**:
- [ ] P0 기능 3-5개 (너무 많으면 범위 축소 고려)
- [ ] P1 기능 3-7개
- [ ] P2/P3 기능 합쳐서 5-10개
- [ ] 각 기능이 페르소나의 목표와 연결됨

---

### Step 4: 측정 가능한 성공 지표 정의 (5분)

**3-5개의 정량적 메트릭**:

```markdown
## 성공 지표
1. **[메트릭명]**: [목표값]
   - 측정 방법: [어떻게 측정할 것인가]
   - 임계값: [최소 허용 값]
```

**예시**:
- **데이터 커버리지**: 80% 이상의 한자에 대해 3개국 발음 제공
- **검색 정확도**: 사용자가 원하는 문자를 상위 5개 결과에서 찾을 확률 > 90%
- **페이지 로딩 속도**: First Contentful Paint < 1.5초
- **도메인 전문가 평가**: 5점 만점에 4.0 이상

**체크리스트**:
- [ ] 모든 메트릭이 숫자로 측정 가능
- [ ] 각 메트릭에 명확한 임계값 설정
- [ ] 측정 방법이 구체적으로 정의됨

---

### Step 5: 범위 경계 정의 (5분)

**명시적으로 포함하지 않을 것**을 정의하세요:

```markdown
## 범위 내 (In Scope)
- ✅ [포함되는 기능/도메인]
- ✅ [포함되는 기능/도메인]

## 범위 외 (Out of Scope)
- ❌ [제외되는 기능/도메인]
- ❌ [제외되는 기능/도메인]
```

**TriHanzi 예시**:
- ✅ 범위 내: 한중일 공통 한자(CJK Unified Ideographs), 기본 발음, 의미
- ❌ 범위 외: 베트남 한자(Chu Nom), 필순 애니메이션, 손글씨 인식

**체크리스트**:
- [ ] 범위 내 항목 3-5개
- [ ] 범위 외 항목 3-5개 (향후 혼란 방지)
- [ ] 도메인 경계가 명확함

## 💡 참고: TriHanzi 실제 PRD 구조

TriHanzi 프로젝트는 암묵적 PRD를 가졌지만, 재구성하면:

- **문제**: 한중일 3개국의 한자가 동일한 Unicode이지만 발음/의미가 다름 → 학습자 혼란
- **페르소나**: 언어 학습자, 교사, 언어학 연구자
- **P0**: 문자 표시, 검색, 3개국 발음
- **P1**: 비교 테이블, False Friends (같은 글자 다른 의미)
- **P2**: 교육 레벨, 데이터 검증 도구
- **성공 지표**: 발음 커버리지 80%, 전문가 평가 A 이상
- **범위**: CJK Unified Ideographs만, 베트남/필순 제외

이 구조를 참고하여 자신의 아이디어를 PRD로 변환하세요.
